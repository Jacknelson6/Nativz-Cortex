# Performance audit + fixes — 2026-04-23

**Author:** Jack + Claude
**Scope:** Cortex load-speed audit, measured + fixed autonomously over
one session.

## TL;DR

Measured `/admin/login` on Lighthouse mobile preset (Moto G Power / Slow
4G / 4× CPU). Baseline was **80**, final is **85–87** (run-to-run noise,
but consistently higher). Payload dropped from 692 KiB → 507 KiB.

| Metric | Baseline | Final | Δ |
|---|---|---|---|
| Performance score | 80 | 85–87 | +5–7 |
| LCP | 5.2 s | 4.0–4.3 s | **−0.9 to −1.2 s** |
| TBT | 70 ms | 10–30 ms | **−40 to −60 ms** |
| Speed Index | 1.5 s | 1.1–1.2 s | −0.3 s |
| Total bytes | 692 KiB | 507 KiB | **−185 KiB** |
| CLS | 0 | 0 | — |

On Lighthouse **desktop** preset (no throttling) the same page scores
**99** — confirming the app itself is fast and the mobile cost is real
mobile-user cost (network + CPU throttling).

## What shipped

### 1. Hero asset diet (`cfe2884` on `main`)

`/admin/login` was pulling **both** `nativz-logo.png` (100 KB, 2060×1113)
**and** `nativz-logo.svg` (134 KB — a fake-vector SVG wrapping a base64
embed of the same 2060×1113 PNG). The logo is displayed at ≤140 px wide.

- Regenerated both as a 400×216 PNG (14 KB) + a 400-wide fake-vector SVG
  wrapper (19 KB). Same callers, same filenames, drop-in replacement.
- `anderson-logo.png` (117 KB, 741×547) → 400-wide (56 KB). Also a
  drop-in replacement at the same filename.
- **Savings: ~162 KB on every page load that hits these assets.**

The "fake-vector" pattern — `<svg><image href="data:image/png;base64,…"/></svg>`
— is common output from design tools (Figma "Save as SVG" with an
embedded PNG). It defeats every optimization in the stack: Next/Image
doesn't touch SVGs, so the raw base64 ships as-is. Worth grepping for
`xlink:href="data:image` / `href="data:image` in any future asset
additions.

### 2. Font diet (same commit)

`app/layout.tsx` loaded **5 Google Font families** × many weights,
plus Rubik a second time as a `--font-geist-sans` legacy alias that
was never actually referenced anywhere in the app.

Repo-wide audit (`grep -c font-light`, `font-extrabold`, `font-black`)
found:
- `font-extrabold` / `font-black` — **0 usages**
- `font-light` — **3 usages** (drop to 400 fallback; imperceptible)
- `font-medium` / `font-semibold` / `font-bold` — keep

Changes:
- Dropped weight `300` from every family.
- Dropped weight `800` from Jost + Sora.
- Killed the `legacySansAlias` duplicate Rubik load entirely.
- Result: ~5 fewer woff2 requests on first paint per page.

### 3. Lazy-load `@react-pdf/renderer` (same commit)

`components/onboard/pdf-download-button.tsx` was eagerly importing
`@react-pdf/renderer` + `./strategy-pdf` at module scope. PDF engine
is ~500 KB of JS only needed when the user clicks "Download PDF".

Moved both into the click handler via dynamic `import()`, matching the
pattern already in place for `components/results/export-pdf-button.tsx`
and `components/audit/audit-export-pdf-button.tsx`.

### 4. 25 missing loading.tsx skeletons (same commit)

Audit: 113 page.tsx files, **72** had `loading.tsx`. Of the 41 missing,
16 were redirect stubs (no loading needed). The other **25 were real
pages that rendered blank while Next fetched server data** — the
classic "click a link, screen goes white for a second" feeling.

Added skeletons for:
- All 10 `/admin/clients/[slug]/settings/*` subpages
- `/admin/search/[id]/processing` + `/subtopics` (admin + portal)
- `/admin/competitor-intelligence/watch` + `/reports` + `/reports/new`
- `/admin/competitor-tracking/tiktok-shop/*` routes
- `/admin/ad-creatives/batches/[batchId]` + v2 variant
- `/admin/onboarding/[id]` + `/email-templates`
- `/admin/nerd/api/[section]`
- `/admin/clients/[slug]/contract`

Extracted 3 tuned skeletons into `components/ui/page-shell-skeleton.tsx`:

- `<PageShellSkeleton tiles grid="cards"|"rows">` — generic list view
- `<SettingsShellSkeleton />` — side-nav + form layout
- `<ProcessingShellSkeleton />` — progress bar + stage list

Each matches the real layout so the swap-in doesn't jank.

## What I didn't touch (and why)

### Mobile LCP floor ≈ 4 s on `'use client'` pages

Even after asset + font trimming, the login page's LCP is ~4 s on mobile.
Breakdown:
- FCP is 1.1 s (HTML + CSS paint).
- LCP happens at ~4 s when the React bundle is parsed, executed, and
  the client page hydrates.

This is **React hydration cost**, not network. The login page is
`'use client'` because it uses `useBrandMode()` for Anderson vs Nativz
branching. To fix properly, it needs splitting into:
- A server-rendered shell (correct colours + logo + hero text) that
  reads `data-brand-mode` from `<html>`, already set by root layout.
- A small client island for the form handler + brand toggle button.

**Not done this session** because visual regression risk is real —
brand-mode branching touches colours/gradients/type on every element.
Worth a follow-up session with Playwright screenshot diffing in the
loop.

### Heavy component lazy-loading

The heaviest pages by chunk count:

| Page | Chunks | Note |
|---|---|---|
| `/admin/search/[id]` | 26 | Imports ~25 report sections eagerly |
| `/admin/clients/[slug]/settings/info` | 21 | Big form |
| `/admin/analyze-social/[id]` | 19 | Audit report |
| `/admin/presentations/[id]/present` | 19 | Presenter mode |

For `/admin/search/[id]/results-client.tsx` specifically, there's a
real win available: below-the-fold sections (BrandApplication,
SourceBrowser, ActivityChart) could be behind `<Suspense>` + lazy.
**Not done** because I couldn't auth-gate Lighthouse to measure the
actual mobile cost, and eyeballing "what's below the fold" depends on
search content length. Worth doing with a Playwright + `sec-ch-viewport`
harness.

### Data-fetching audit

Audited the top 5 RSC pages (`dashboard`, `pipeline`, `strategy-lab`,
`clients`, `knowledge`). Hot paths already use `Promise.all` where
independent fetches can parallelise. `strategy-lab` has a longer
waterfall (9 sequential awaits) but most of its awaits are dependency
chains (searchParams → activeClient → scope-specific fetch). No safe
blind refactor.

### Link prefetching

Didn't find raw `<a href="/admin/…">` tags in hot nav code — the sidebar
and top bar both use `<Link>`, which auto-prefetches in Next.js 15.
Nothing to fix.

### Raw `<img>` tags (CLS risk)

Found ~20 raw `<img>` tags. All for:
- Static agency SVG lockups (Anderson) — vector, 14 KB, low risk
- Remote avatars / thumbnails (user, post, client logos) — fixing
  these would require verifying every Supabase / CDN domain is in
  `next.config.ts` `remotePatterns`, and behaviour differs for
  user-supplied URLs. Out of scope for an autonomous pass.

```bash
# Mobile preset (matches this doc's numbers)

  --chrome-flags="--headless=new --no-sandbox" \
  --output=json --output-path=/tmp/lh.json \
  --quiet --only-categories=performance

# Desktop (for the 99 number)
npx lighthouse http://localhost:3100/admin/login \
  --chrome-flags="--headless=new --no-sandbox" \
  --preset=desktop \
  --output=json --output-path=/tmp/lh-desktop.json \
  --quiet --only-categories=performance
```

(ephemeral; won't survive reboots).
